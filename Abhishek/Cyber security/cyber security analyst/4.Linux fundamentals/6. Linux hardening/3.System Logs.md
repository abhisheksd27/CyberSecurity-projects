## Why Logs Matter (Both Defensively and Offensively)

System logs are a two-sided coin. Defenders use them to detect intrusions and audit activity. As a penetration tester, you use them to:

- Understand what's running on the system and how it behaves
- Find credentials or sensitive data inadvertently written to logs
- Identify other users, services, and scheduled tasks
- Check whether your own activities left traces (useful for operational security)
- Find misconfigurations and vulnerabilities exposed through logged errors

### 1. Kernel Logs — `/var/log/kern.log`

Contains low-level system events: hardware driver messages, system calls, kernel module activity, hardware errors.

From a pentest perspective:

- Shows **outdated or vulnerable drivers** that could be exploited
- **System crash** patterns that suggest instability or DoS potential
- **Unusual syscalls** that could indicate malware already on the system
- Hardware errors that point to resource constraints

Example kernel lines from the syslog in the document:

```
kernel: [  138.303596] ata3.00: exception Emask 0x0 SAct 0x0 SErr 0x0 action 0x6 frozen
kernel: [  367.543975] EXT4-fs (sda1): re-mounted. Opts: errors=remount-ro
```

The first is a disk error. The second shows the filesystem was remounted — possibly after a crash recovery. Both are worth noting.

### 2. System Logs — `/var/log/syslog`

A general-purpose log that captures service starts/stops, cron jobs, login attempts, system reboots, and kernel messages. It's the broadest view of what's happening on a system.

Reading the example syslog from the document:

```
Feb 28 15:00:01 server CRON[2715]: (root) CMD (/usr/local/bin/backup.sh)
```

→ A cron job running as **root** executes `/usr/local/bin/backup.sh`. If that script is world-writable, that's a privilege escalation path.

```
Feb 28 15:04:22 server sshd[3010]: Failed password for htb-student from 10.14.15.2 port 50223 ssh2
Feb 28 15:07:19 server sshd[3010]: Accepted password for htb-student from 10.14.15.2 port 50223 ssh2
```

→ Failed login followed by a successful one from the same IP — brute force that succeeded.

```
Feb 28 15:06:43 server apache2[2904]: 127.0.0.1 - - "GET /index.html HTTP/1.1" 200 13484
```

→ Apache is running and serving requests. Note the 200 status.

### 3. Authentication Logs — `/var/log/auth.log`

The most valuable log for security analysis. Specifically focused on authentication — SSH logins, sudo usage, PAM sessions, su attempts.

Dissecting the example `auth.log` entry by entry:

```
sshd[5678]: Accepted publickey for admin from 10.14.15.2 port 43210 ssh2: RSA SHA256:+KjEzN2cVhIW/...
```

→ Admin logged in via SSH public key from `10.14.15.2`. The key fingerprint is logged — useful for tracking which key was used.

```
sudo: admin : TTY=pts/1 ; PWD=/home/admin ; USER=root ; COMMAND=/bin/bash
```

→ Admin immediately ran `sudo /bin/bash` — spawning a root shell. This is highly suspicious in a real environment.

```
sudo: admin : TTY=pts/1 ; PWD=/home/admin ; USER=root ; COMMAND=/usr/bin/apt-get install netcat-traditional
```

→ Installed `netcat-traditional` as root. Classic attacker tool. This is a huge red flag — a defender seeing this should treat it as a confirmed compromise.

```
sshd[5678]: Disconnected from 10.14.15.2 port 43210 [preauth]
```

→ Disconnected at the pre-authentication stage — could indicate a port scan or probe.

```
kernel: [  778.941871] firewall: unexpected traffic allowed on port 22
```

→ Firewall logged something anomalous on port 22.

```
CRON[2345]: pam_unix(cron:session): session opened for user root by (uid=0)
CRON[2345]: pam_unix(cron:session): session closed for user root
```

→ A cron job ran as root and completed. Worth cross-referencing with the actual crontab.

### 4. Application Logs

Each major service writes its own logs. Key locations:

|Service|Log path|
|---|---|
|Apache|`/var/log/apache2/access.log`, `/var/log/apache2/error.log`|
|Nginx|`/var/log/nginx/access.log`, `/var/log/nginx/error.log`|
|MySQL|`/var/log/mysql/mysql.log`|
|PostgreSQL|`/var/log/postgresql/postgresql-version-main.log`|
|OpenSSH|`/var/log/auth.log` (Ubuntu) or `/var/log/secure` (CentOS/RHEL)|
|Systemd|`/var/log/journal/` (read with `journalctl`)|

The access log example from the document is a standout:

```
2023-03-07T10:15:23+00:00 servername privileged.sh: htb-student accessed /root/hidden/api-keys.txt
```

A regular user (`htb-student`) used a script (`privileged.sh`) — likely SUID or sudo-allowed — to read a file in `/root/hidden/`. The file is called `api-keys.txt`. This is exactly the kind of log entry that reveals a privilege escalation has already occurred.

### 5. Security Logs

Generated by dedicated security tools:

|Tool|Log location|
|---|---|
|`fail2ban`|`/var/log/fail2ban.log`|
|`ufw`|`/var/log/ufw.log`|
|`auditd`|`/var/log/audit/audit.log`|

These logs tell you what security controls are active, what they've been triggered by, and whether your own pentest activities are being detected.

## Analyzing Logs — Command Line Tools

bash

```bash
# View the last 50 lines of a log in real time
tail -f -n 50 /var/log/auth.log

# Search for failed SSH attempts
grep "Failed password" /var/log/auth.log

# Search for sudo usage
grep "sudo" /var/log/auth.log

# Find all successful logins
grep "Accepted" /var/log/auth.log

# Extract just the IP addresses from failed logins
grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -rn

# Search syslog for a specific time window
grep "Feb 28 15:" /var/log/syslog

# Check for cron jobs run as root
grep "CRON.*root" /var/log/syslog
```

## HTB / Pentest Cheat Sheet for Logs

Once you have shell access on a target, these are high-value log reads:

bash

```bash
# Who has been logging in?
cat /var/log/auth.log | grep "Accepted"

# Any failed logins (shows attacker IPs or brute force)?
cat /var/log/auth.log | grep "Failed"

# What has root been doing via cron?
grep "CRON" /var/log/syslog | grep root

# What commands have been run with sudo?
grep "sudo" /var/log/auth.log | grep "COMMAND"

# Any web server access logs — what endpoints exist?
cat /var/log/apache2/access.log | awk '{print $7}' | sort | uniq -c | sort -rn

# Check if fail2ban is running and what it's blocking
cat /var/log/fail2ban.log | tail -20

# Check kernel for driver issues or anomalies
dmesg | grep -i error
```

The key habit: **logs tell the story of what happened on a system.** When you land on a target, reading the logs isn't just defensive research — it reveals what other users have done, what services are active, and often, what sensitive data has been touched.
