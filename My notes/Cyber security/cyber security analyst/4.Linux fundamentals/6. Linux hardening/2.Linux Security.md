## Part 1: Linux Security Fundamentals

Security isn't a one-time setup — it's an ongoing process. The document covers several layers of protection you should have in place.

### 1. Keep the System Updated

bash

```bash
apt update && apt dist-upgrade
```

`apt update` refreshes the package list. `apt dist-upgrade` upgrades all packages including the kernel. This is the single most impactful thing you can do — many exploits target known, unpatched vulnerabilities. Note that some kernel updates require a manual reboot or won't apply automatically.

### 2. Firewall & iptables

If network-level firewalling isn't configured (e.g., on a cloud provider's security group), you use `iptables` or `ufw` directly on the host to control which ports accept traffic. This is your first line of defence.

### 3. Hardening SSH

SSH is a prime target. Best practices:

- **Disable password login** → use key-based authentication only (`PasswordAuthentication no`)
- **Disable root login** → (`PermitRootLogin no`) — attackers always try `root` first
- Use **fail2ban** — it monitors failed login attempts and bans IPs that exceed a threshold

### 4. Principle of Least Privilege

Users should only have the permissions they absolutely need. Instead of giving a user full `sudo` access, you can lock it down to specific commands in `/etc/sudoers`:

bash

```bash
username ALL=(ALL) /usr/bin/specific-command
```

This way, even if that account is compromised, the attacker's privilege escalation options are limited.

### 5. Periodic System Auditing

Check for:

- Out-of-date kernel versions
- World-writable files (`find / -perm -o+w`)
- Misconfigured cron jobs (run as root?)
- SUID/SGID binaries that shouldn't be executable as other users

### 6. SELinux and AppArmor

These are **mandatory access control (MAC)** systems — a layer beyond standard Unix permissions.

- **SELinux** — labels every process, file, and object. Policy rules control which labeled process can access which labeled file. Very granular (can control who can _append_ vs _read_ a file).
- **AppArmor** — profile-based, simpler to configure than SELinux, used in Ubuntu by default.

Both operate at the **kernel level**, so even if an attacker gets inside a process, they're constrained by the policy.

## Part 2: TCP Wrappers

TCP Wrappers are a host-based access control mechanism that sits in front of services and decides whether to allow a connection based on the client's IP or hostname — _before_ the service even processes the request.

### The Two Config Files

**`/etc/hosts.allow`** — whitelist rules. If a rule here matches, access is granted immediately without checking `hosts.deny`.

**`/etc/hosts.deny`** — blacklist rules. If no rule in `hosts.allow` matched, this file is checked. If a match is found here, the connection is refused.

If neither file has a matching rule → the connection is **allowed by default**.

### Rule Syntax

```
service : source
```

Some examples from the document:

bash

```bash
# /etc/hosts.allow
sshd : 10.129.14.0/24        # Allow SSH from entire /24 subnet
ftpd : 10.129.14.10          # Allow FTP from one specific IP
telnetd : .inlanefreight.local  # Allow Telnet from entire domain (note the leading dot)
```

bash

```bash
# /etc/hosts.deny
ALL : .inlanefreight.com     # Block ALL services from this domain
sshd : 10.129.22.22          # Block SSH from this specific IP
ftpd : 10.129.22.0/24        # Block FTP from entire subnet
```

The leading dot (`.inlanefreight.local`) means "any host ending in that domain" — it's a wildcard for subdomains.

`ALL` as the service name applies the rule to every service, making it a catch-all deny.

### Critical Things to Remember

|Point|Details|
|---|---|
|Order matters|First matching rule wins — no further rules are checked|
|Not a firewall|TCP Wrappers control access to **services**, not **ports**. A firewall like iptables blocks at the network/port level.|
|Works at service layer|The check happens when a connection reaches the service, not at the network boundary|
|Default allow|If no rule matches in either file, the connection is allowed|

## Part 3: Additional Security Tools Mentioned

|Tool|Purpose|
|---|---|
|`fail2ban`|Bans IPs after repeated failed login attempts|
|`Snort`|Intrusion detection / prevention system (IDS/IPS)|
|`chkrootkit`|Scans for known rootkits|
|`rkhunter`|Rootkit Hunter — checks for rootkits, backdoors, local exploits|
|`Lynis`|Full system security audit tool — produces a hardening report|
|`SELinux` / `AppArmor`|Mandatory access control at kernel level|

## HTB / Pentest Takeaways

This module is building your defensive knowledge, which also sharpens your offensive eye. When assessing a Linux target:

- **SSH with password login enabled** → brute-force is viable, fail2ban may or may not be running
- **Root SSH login allowed** → misconfiguration, high-value target
- **SUID binaries** → a classic privilege escalation path (`find / -perm -4000 2>/dev/null`)
- **World-writable files / misconfigured cron jobs** → another privesc avenue
- **TCP Wrappers present** → your IP might get blocked mid-engagement if you trigger a deny rule; worth checking `/etc/hosts.deny` if you get shell access
- **No SELinux/AppArmor** → once inside, lateral movement and file access is less restricted
