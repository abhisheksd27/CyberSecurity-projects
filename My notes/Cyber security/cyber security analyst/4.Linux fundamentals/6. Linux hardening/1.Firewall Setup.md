## The Big Picture: How Linux Firewalling Works

Linux doesn't implement firewalling in a userspace tool — it happens inside the **kernel** via the **Netfilter framework**. Netfilter provides hooks at key points in the network stack where packets can be intercepted and acted upon. `iptables` is simply the command-line interface you use to write rules into those kernel hooks.

The evolution of Linux firewall tools:

|Tool|Era|Notes|
|---|---|---|
|`ipfwadm`|Early Linux|Oldest, deprecated|
|`ipchains`|Pre-2.4 kernel|Replaced ipfwadm|
|`iptables`|2000, kernel 2.4+|De facto standard for years|
|`nftables`|Modern|Better performance, different syntax|
|`ufw`|Ubuntu-focused|Simplified frontend over iptables|
|`firewalld`|RHEL/CentOS|Dynamic, zone-based management|

## The iptables Architecture

Everything in iptables flows through a hierarchy: **Tables → Chains → Rules → Targets**. Think of it like a sorting facility — packets arrive, get routed to the right department (table), pass through the right checkpoint (chain), get evaluated against rules, and finally get acted upon (target).---

## Tables in Detail

Each table handles a different _kind_ of work on a packet:

**filter** — the default table. This is where you allow or block traffic. Most rules live here. Its three chains map directly to traffic direction: `INPUT` (arriving at this machine), `OUTPUT` (leaving this machine), `FORWARD` (passing through this machine to somewhere else — relevant when the host acts as a router).

**nat** — used for Network Address Translation. `PREROUTING` modifies the destination IP _before_ the kernel's routing decision (useful for port forwarding). `POSTROUTING` modifies the source IP _after_ routing (useful for masquerading private IPs as a public IP).

**mangle** — modifies packet header fields like TTL, TOS bits, or Netfilter marks. Used for advanced traffic shaping and QoS. Rarely needed for basic firewalling.

**raw** — earliest point of interception, before connection tracking. Used to mark packets that should bypass the connection tracking system (`NOTRACK`).

## How Packet Flow Works Through Chains

The packet path in plain English: an incoming packet hits `PREROUTING` first, then the kernel makes a routing decision — if the packet is destined for this machine it goes to `INPUT`, if it needs to be forwarded to another host it goes to `FORWARD`. Packets generated locally leave via `OUTPUT` then `POSTROUTING`.

## Rules and Matches — Building Commands

Every `iptables` rule follows this structure:

```bash
iptables -A <CHAIN> <matches> -j <TARGET>
```

`-A` = append to chain. Matches narrow down which packets the rule applies to. `-j` = jump to this target if matched.

### The 10 Practice Tasks — Commands Explained

**1. Block incoming traffic on port 8080**

```bash
sudo iptables -A INPUT -p tcp --dport 8080 -j DROP
```

**2. Allow incoming traffic on port 8080**

```bash
sudo iptables -A INPUT -p tcp --dport 8080 -j ACCEPT
```

Note: rule order matters. If a DROP rule for 8080 already exists higher in the chain, this ACCEPT won't be reached. Use `-I` (insert at top) instead of `-A` (append) if needed.

**3. Block traffic from a specific IP**

```bash
sudo iptables -A INPUT -s 10.129.22.22 -j DROP
```

**4. Allow traffic from a specific IP**

```bash
sudo iptables -A INPUT -s 10.129.14.10 -j ACCEPT
```

**5. Block traffic based on protocol (e.g., ICMP / ping)**

```bash
sudo iptables -A INPUT -p icmp -j DROP
```

**6. Allow traffic based on protocol (e.g., UDP)**

```bash
sudo iptables -A INPUT -p udp -j ACCEPT
```

**7. Create a new (user-defined) chain**

```bash
sudo iptables -N MY_CHAIN
```

**8. Forward traffic to a specific chain**

```bash
sudo iptables -A INPUT -p tcp --dport 8080 -j MY_CHAIN
```

This sends matching packets into your custom chain for further processing.

**9. Delete a specific rule**

First list rules with line numbers:

```bash
sudo iptables -L INPUT --line-numbers
```

Then delete by number:

```bash
sudo iptables -D INPUT 3
```

**10. List all existing rules**

```bash
sudo iptables -L -v -n
```

`-v` = verbose (shows packet/byte counts), `-n` = numeric (don't resolve IPs to hostnames, faster).

## Key Matches Reference

|Match|Example|What it does|
|---|---|---|
|`-p tcp`|`-p tcp`|Match TCP packets|
|`--dport`|`--dport 443`|Match destination port|
|`--sport`|`--sport 1024:65535`|Match source port range|
|`-s`|`-s 192.168.1.0/24`|Match source IP / subnet|
|`-d`|`-d 10.0.0.5`|Match destination IP|
|`-m state`|`-m state --state ESTABLISHED`|Match connection state|
|`-m multiport`|`-m multiport --dports 80,443,8080`|Match multiple ports at once|
|`-m mac`|`-m mac --mac-source AA:BB:CC:DD:EE:FF`|Match by MAC address|

## DROP vs REJECT — Important Distinction

`DROP` silently discards the packet. The sender gets no response and has to wait for a timeout. From a stealth standpoint this is preferred — the attacker doesn't know if a firewall is present or the port just doesn't exist.

`REJECT` drops the packet but sends back an ICMP error (`port-unreachable` or `tcp-reset`). The sender knows immediately the connection was refused. Faster for legitimate users, but more informative to attackers.

## HTB / Pentest Takeaways

From an offensive perspective:

- **DROP vs REJECT fingerprinting** — timeout responses suggest DROP rules (firewall), immediate resets suggest REJECT or a closed port. This informs your scanning strategy.
- **Port 8080 open with no firewall rule** → web server likely running, worth probing
- **`iptables -L` readable without sudo** → misconfiguration, you can map the firewall rules as a low-priv user
- **No stateful rules (`-m state --state ESTABLISHED`)** → firewall may not properly handle connection tracking, potentially exploitable for firewall bypass
- **FORWARD chain policies** → tells you if the host is acting as a router/gateway between network segments, high value for pivoting
