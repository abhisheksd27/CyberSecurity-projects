## The Big Picture: Why Remote Desktop Protocols Exist

When a system admin needs to manage a remote machine visually (not just via terminal), they use a **remote desktop protocol**. These protocols let you see and interact with the graphical desktop of a remote computer over a network — like sitting in front of it.

Two main protocols:

- **RDP** (Remote Desktop Protocol) → Windows-focused
- **VNC** (Virtual Network Computing) → Cross-platform, popular on Linux

Think of it like different keys for different buildings — you pick the right protocol for the OS you're managing.

## Part 1: XServer and X11

Before VNC/RDP, Linux has its own native system for graphical interfaces — the **X Window System (X11)**.

![[Pasted image 20260320213829.png]]

**Key idea:** With X11, the _application_ runs on the remote machine, but the _rendering_ (drawing to screen) happens on your local machine. This is the opposite of VNC/RDP where the remote machine renders and sends you an image.

- **Advantage:** Saves bandwidth and CPU on the remote machine
- **Disadvantage:** X11 traffic is **unencrypted by default** — a serious security risk

### X11 Forwarding over SSH

To fix the encryption problem, you tunnel X11 through SSH:

1. Enable it on the server: set `X11Forwarding yes` in `/etc/ssh/sshd_config`
2. Connect with: `ssh -X user@remote /usr/bin/firefox`

The `-X` flag tells SSH to forward X11 display data securely through the encrypted SSH tunnel.

### X11 Security Risks

- **Open X server** → attackers on the same network can capture screenshots using tools like `xwd` and `xgrabsc` without sniffing packets in the traditional sense
- **CVE-2017-2624, 2625, 2626** → vulnerabilities in XOrg Server where weak session keys allowed arbitrary code execution in another user's Xorg session. Affected: Ubuntu, Red Hat, SUSE, and other Unix systems.
- **Ports 6000–6010** are the red flag to watch for during a pentest — an open X server is a significant exposure.

### XDMCP

A sub-protocol used by the **X Display Manager** for managing remote X Window sessions. Key facts:

- Runs over **UDP port 177**
- Allows redirecting entire GUIs (like KDE/GNOME) to a remote client
- **Inherently insecure** — no strong authentication
- Vulnerable to **man-in-the-middle attacks** where an attacker impersonates either party to gain unauthorized server access

## Part 2: VNC

VNC (Virtual Network Computing) is the practical go-to for graphical remote access on Linux. Unlike X11, VNC renders the desktop on the remote machine and streams the image to you — similar to how RDP works on Windows.

![[Pasted image 20260320213850.png]]
### VNC Port Numbering — Important

|Display|Port|
|---|---|
|`:0`|`5900`|
|`:1`|`5901`|
|`:2`|`5902`|
|`:N`|`590N`|

The pattern is `5900 + display number`. So when you run `vncserver` and it assigns `linux:1`, you connect on port **5901**.

### VNC Setup — Step by Step Explained

**1. Install packages**

bash

```bash
sudo apt install xfce4 xfce4-goodies tigervnc-standalone-server -y
```

GNOME is unstable with VNC, so **XFCE4** is used as the lightweight desktop manager.

**2. Set a password**

bash

```bash
vncpasswd
```

Creates a hashed password stored in `~/.vnc/passwd`. This is what clients must provide to connect.

**3. Configure `~/.vnc/xstartup`** This script tells the VNC session what desktop to launch. The key line is:

bash

````bash
/usr/bin/startxfce4
```
This starts the XFCE desktop when a VNC client connects. Without this, you'd get a blank screen.

**4. Configure `~/.vnc/config`**
Sets the display resolution and DPI:
```
geometry=1920x1080
dpi=96
````

**5. Start the server**

bash

```bash
vncserver
```

This assigns `display :1` → listens on **port 5901**.

**6. List running sessions**

bash

```bash
vncserver -list
```

Shows which display numbers are active, their ports, and their PIDs — useful for managing or killing sessions.

## Part 3: Securing VNC with an SSH Tunnel

Raw VNC traffic, while it has basic authentication, can be further secured by tunneling through SSH. The command:

bash

```bash
ssh -L 5901:127.0.0.1:5901 -N -f -l htb-student 10.129.14.130
```

|Flag|Meaning|
|---|---|
|`-L 5901:127.0.0.1:5901`|Forward local port 5901 → remote's localhost:5901|
|`-N`|Don't execute a remote command, just tunnel|
|`-f`|Run in the background|
|`-l htb-student`|Login as this user|

After the tunnel is up, you connect your VNC viewer to **localhost:5901** — your viewer thinks it's talking to your own machine, but the traffic is securely forwarded through the SSH connection to the remote VNC server.

## Summary Table

|Protocol|Transport|Default Port|Renders on|Encryption|Primary Use|
|---|---|---|---|---|---|
|X11|TCP/IP or Unix socket|6000–6009|**Local** machine|None (use SSH -X)|Linux native GUI forwarding|
|XDMCP|UDP|177|Remote|None|Remote GUI session management (insecure)|
|VNC|TCP|5900+|**Remote** machine|Basic (use SSH tunnel)|Cross-platform remote desktop|
|RDP|TCP|3389|Remote|Yes (built-in)|Windows remote desktop|

## HTB / Pentest Takeaways

From a security perspective, this module is teaching you to:

1. **Identify open X11 ports (6000–6010)** on a target — these can leak graphical data
2. **XDMCP on UDP 177** → weak, no encryption, MITM-able
3. **VNC on 5900–5910** → look for open VNC servers, brute-force passwords, or intercept unencrypted sessions
4. **SSH -X forwarding** → useful for running GUI tools on a remote machine during a pentest (e.g., launching a browser through an SSH session to browse an internal web app)
